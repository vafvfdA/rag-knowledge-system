## RedisTemplate序列化策略
`RedisTemplate`的序列化策略在`RedisConfig`类中定义，直接影响缓存数据的存储格式和可读性。

### 序列化策略分析
```java
@Configuration
public class RedisConfig {

    @Bean
    public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory connectionFactory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(connectionFactory);
        template.setKeySerializer(new StringRedisSerializer());
        template.setValueSerializer(new GenericJackson2JsonRedisSerializer());
        return template;
    }
}
```

**关键配置：**
- **Key序列化器 (keySerializer)**: 使用`StringRedisSerializer`，将键（Key）序列化为UTF-8编码的字符串。这保证了键在Redis中是可读的文本。
- **Value序列化器 (valueSerializer)**: 使用`GenericJackson2JsonRedisSerializer`，将值（Value）序列化为JSON格式的字符串。

### JSON与JDK序列化的选择
本项目选择了JSON序列化（`GenericJackson2JsonRedisSerializer`）而非JDK原生序列化，原因如下：
1. **可读性**: JSON是纯文本格式，在Redis客户端中可直接查看和理解数据内容，便于调试和运维。
2. **跨语言兼容性**: JSON是通用的数据交换格式，其他语言的应用程序也能轻松读取和写入缓存数据。
3. **避免类版本问题**: JDK序列化要求反序列化时的类结构与序列化时完全一致，而JSON序列化对类结构变化更宽容。

## 连接工厂与线程安全性
### 连接工厂配置
项目使用了Spring Data Redis的默认连接工厂。虽然代码中没有显式配置`LettuceConnectionFactory`，但Spring Boot会根据类路径上的依赖自动配置。`Lettuce`是一个可扩展的、线程安全的Redis客户端，支持同步、异步和响应式操作。

### 线程安全性
`RedisTemplate`本身是线程安全的。`RedisConfig`中通过`@Bean`注解定义的`redisTemplate`是一个单例（Singleton）Bean，由Spring容器管理。这意味着：
- 所有需要`RedisTemplate`的地方都注入的是同一个实例。
- `RedisTemplate`内部封装了对底层连接的管理，其操作方法（如`opsForValue().set()`）是线程安全的。
- 开发人员无需担心在多线程环境下使用`RedisTemplate`会导致数据竞争。

## 缓存过期时间(TTL)设置原则
缓存过期时间（TTL）的设置遵循“业务需求驱动”和“资源优化”的原则，具体体现在不同的业务场景中。

### TTL设置原则
1. **与业务逻辑匹配**: TTL的长度应与缓存数据的有效期一致。例如，会话相关的token缓存时间与JWT的过期时间紧密相关。
2. **防止内存泄漏**: 设置合理的TTL可以自动清理过期数据，避免Redis内存被无效数据占满。
3. **平衡性能与数据一致性**: TTL过长可能导致数据陈旧，过短则会增加数据库查询压力。

### 代码中的TTL实践
- **会话历史记录**: 在`RedisRepository`中，会话历史记录的TTL设置为7天。
  ```java
  redisTemplate.opsForValue().set("conversation:" + conversationId, objectMapper.writeValueAsString(messages), Duration.ofDays(7));
  ```
- **组织标签缓存**: 在`OrgTagCacheService`中，用户组织标签的TTL设置为24小时。
  ```java
  private static final long CACHE_TTL_HOURS = 24;
  redisTemplate.expire(key, CACHE_TTL_HOURS, TimeUnit.HOURS);
  ```
- **Token缓存**: 在`TokenCacheService`中，token的TTL根据JWT的过期时间动态计算，并额外增加5分钟的缓冲时间。
  ```java
  long ttlSeconds = (expireTimeMs - System.currentTimeMillis()) / 1000 + 300; // 多5分钟缓冲
  redisTemplate.opsForValue().set(key, tokenInfo, ttlSeconds, TimeUnit.SECONDS);
  ```

## 缓存击穿、穿透、雪崩应对策略
项目通过`RedisRepository`和相关服务类的实现，间接体现了对缓存常见问题的应对策略。

### 缓存击穿 (Cache Breakdown)
**问题**: 某个热点Key在过期的瞬间，大量并发请求同时涌入数据库。
**应对策略**: 本项目未显式使用互斥锁（Mutex）或逻辑过期（Logical Expiration）来解决击穿问题。但在`TokenCacheService`中，通过将token信息缓存并设置TTL，结合`isTokenValid`方法检查缓存，减少了对数据库的直接查询，一定程度上缓解了击穿风险。

### 缓存穿透 (Cache Penetration)
**问题**: 查询一个数据库中不存在的数据，导致每次请求都绕过缓存。
**应对策略**: 项目代码中未直接体现布隆过滤器（Bloom Filter）或缓存空值的策略。但`RedisRepository`在获取会话历史时，会对空值进行处理，返回一个空的`ArrayList`，这可以防止上层逻辑因`null`值而产生异常。
```java
return json == null ? new ArrayList<>() : objectMapper.readValue(json, ...);
```

### 缓存雪崩 (Cache Avalanche)
**问题**: 大量缓存Key在同一时间过期，导致数据库瞬间压力剧增。
**应对策略**: 项目通过为不同业务设置不同的TTL来避免雪崩。例如，会话历史缓存7天，组织标签缓存24小时，token缓存时间动态计算。这种差异化的过期时间策略，有效分散了缓存失效的时间点。

## 缓存Key命名规范与分层策略
项目遵循了清晰、一致的缓存Key命名规范，采用“域:子域:标识符”的分层结构。

### 命名规范
1. **前缀化**: 所有Key都使用有意义的前缀，明确标识其所属的业务域。
2. **冒号分隔**: 使用冒号（`:`）作为层级分隔符，提高可读性。
3. **具体标识**: Key的最后部分通常是具体的业务标识符（如用户名、ID）。

### 分层策略与代码示例
```java
// TokenCacheService 中的Key命名
private static final String TOKEN_PREFIX = "jwt:valid:"; // 域:子域
private static final String USER_TOKENS_PREFIX = "jwt:user:"; // 域:子域
// 实际Key: "jwt:valid:abc123" 或 "jwt:user:123:tokens"

// OrgTagCacheService 中的Key命名
private static final String USER_ORG_TAGS_KEY_PREFIX = "user:org_tags:"; // 域:子域
private static final String USER_PRIMARY_ORG_KEY_PREFIX = "user:primary_org:"; // 域:子域
// 实际Key: "user:org_tags:alice" 或 "user:primary_org:alice"
```

**优点**:
- **可维护性**: 通过前缀可以轻松地在Redis中查找和管理某一类缓存。
- **避免冲突**: 不同业务的Key不会相互覆盖。
- **语义清晰**: Key的结构直接反映了其业务含义。

## 业务场景下的缓存最佳实践
### 用户会话缓存 (TokenCacheService)
`TokenCacheService`是用户会话管理的核心，其最佳实践包括：
1. **状态管理**: 使用Redis缓存JWT token的状态，实现主动登出和token吊销。
2. **多维度缓存**: 不仅缓存token本身，还通过`USER_TOKENS_PREFIX`为每个用户维护一个token集合，方便批量管理。
3. **黑名单机制**: 使用`BLACKLIST_PREFIX`实现token黑名单，确保已注销的token无法再被使用。
4. **TTL动态计算**: 缓存时间与JWT过期时间联动，并增加缓冲期，保证用户体验。

```java
// 缓存token信息
redisTemplate.opsForValue().set(key, tokenInfo, ttlSeconds, TimeUnit.SECONDS);
// 将token加入用户集合
addTokenToUser(userId, tokenId, expireTimeMs);
// 主动将token加入黑名单
redisTemplate.opsForValue().set(key, System.currentTimeMillis(), ttlSeconds, TimeUnit.SECONDS);
```

### 权限数据缓存 (OrgTagCacheService)
`OrgTagCacheService`用于缓存用户的组织标签权限，其最佳实践包括：
1. **提升性能**: 将频繁查询的权限数据缓存，避免每次请求都访问数据库。
2. **复杂逻辑缓存**: `getUserEffectiveOrgTags`方法不仅缓存直接标签，还递归计算并缓存所有父标签，将复杂的权限计算结果缓存起来。
3. **缓存失效管理**: 提供`deleteUserOrgTagsCache`方法，在用户权限变更时主动删除相关缓存，保证数据一致性。
4. **兜底策略**: 在获取有效标签时，即使发生异常，也确保返回包含`DEFAULT_ORG_TAG`的列表，保证系统基本可用性。

```java
// 缓存用户直接拥有的组织标签
redisTemplate.opsForList().rightPushAll(key, orgTags.toArray());
// 缓存计算后的有效标签集合（包含父标签）
redisTemplate.opsForList().rightPushAll(cacheKey, result.toArray());
// 主动删除缓存
redisTemplate.delete(orgTagsKey);
```
