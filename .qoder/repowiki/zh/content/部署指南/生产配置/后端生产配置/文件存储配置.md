## MinIO核心配置项解析

在 `application-docker.yml` 和 `application-dev.yml` 配置文件中，MinIO 的核心配置项定义了文件存储服务的基本连接参数和行为。这些配置项通过 Spring 的 `@Value` 注解注入到 `MinioConfig` 类中，确保服务能够正确连接到 MinIO 对象存储。

### 核心配置项说明

**MinIO 配置项**
- **endpoint**: 指定 MinIO 服务器的访问地址。在 `application-docker.yml` 中为 `http://localhost:19000`，在 `application-dev.yml` 中为 `http://localhost:9000`。
- **accessKey**: 访问 MinIO 的访问密钥，用于身份验证。Docker 环境使用 `aHZtaNPEzLcERm9uCY2F`，开发环境使用 `minioadmin`。
- **secretKey**: 访问 MinIO 的密钥，与 accessKey 配对使用。Docker 环境使用 `5lhyVdLKfMfYXtmdVsulLYpgCRx5EO1EfIWAtJdq`，开发环境使用 `minioadmin`。
- **bucketName**: 指定默认的存储桶名称，本项目中为 `uploads`。
- **publicUrl**: 提供 MinIO 服务的公共访问地址，用于生成可公开访问的预签名 URL。在两个环境中均设置为与 endpoint 相同的值。

这些配置项共同构成了与 MinIO 服务通信的基础，确保应用程序能够安全、可靠地进行文件的上传、下载和管理。

## MinioClient初始化与Spring集成

`MinioConfig` 类是 Spring 配置类，负责初始化 `MinioClient` 实例并将其注册为 Spring 容器中的 Bean，从而实现与 Spring 框架的无缝集成。

### 配置类实现

```java
@Configuration
public class MinioConfig {

    @Value("${minio.endpoint}")
    private String endpoint;

    @Value("${minio.accessKey}")
    private String accessKey;

    @Value("${minio.secretKey}")
    private String secretKey;

    @Value("${minio.publicUrl}")
    private String publicUrl;

    @Bean
    public MinioClient minioClient() {
        return MinioClient.builder()
                .endpoint(endpoint)
                .credentials(accessKey, secretKey)
                .build();
    }

    @Bean
    public String minioPublicUrl() {
        return publicUrl;
    }
}
```

该类通过 `@Configuration` 注解声明为配置类，并使用 `@Bean` 注解定义了两个 Bean：
- `minioClient()`: 创建并返回一个 `MinioClient` 实例，该实例使用从配置文件中读取的 endpoint、accessKey 和 secretKey 进行初始化。
- `minioPublicUrl()`: 返回 MinIO 的公共访问地址，供其他服务组件注入使用。

通过这种方式，任何需要与 MinIO 交互的服务都可以通过 `@Autowired` 注解直接注入 `MinioClient` 或 `minioPublicUrl`，实现了依赖注入和配置解耦。

## 文件上传与元数据管理

文件上传功能由 `UploadService` 类实现，该服务处理文件分片上传、状态管理、元数据存储和最终合并等核心逻辑。

### 文件上传流程

文件上传采用分片上传机制，以支持大文件的高效传输和断点续传。主要流程如下：

1. **分片上传**: 客户端将文件分割为多个分片，逐个调用 `uploadChunk` 方法上传。
2. **状态检查**: 服务端检查该文件的 MD5 值是否已存在，若不存在则创建新的 `FileUpload` 记录。
3. **分片存储**: 每个分片以 `chunks/{fileMd5}/{chunkIndex}` 的路径存储在 MinIO 中。
4. **元数据管理**: 分片的元数据（如 chunkMd5、storagePath）存储在 `ChunkInfo` 实体中，并通过 Redis 缓存已上传的分片信息以提高性能。
5. **文件合并**: 当所有分片上传完成后，触发合并操作，将分片组合成完整文件。

### 核心实体类

#### FileUpload 实体
```java
@Data
@Entity
@Table(name = "file_upload")
public class FileUpload {
    @Id
    private Long id;
    private String fileMd5;
    private String fileName;
    private long totalSize;
    private int status; // 0-上传中 1-已完成
    private String userId;
    private String orgTag;
    private boolean isPublic = false;
    private LocalDateTime createdAt;
    private LocalDateTime mergedAt;
}
```
该实体用于记录文件上传的全局信息，包括文件的唯一标识（fileMd5）、名称、大小、状态、所属用户和组织标签等。

#### ChunkInfo 实体
```java
@Data
@Entity
@Table(name = "chunk_info")
public class ChunkInfo {
    @Id
    private Long id;
    private String fileMd5;
    private int chunkIndex;
    private String chunkMd5;
    private String storagePath;
}
```
该实体用于记录每个文件分片的详细信息，确保分片的完整性和可追溯性。

## 安全配置与最佳实践

尽管当前配置将敏感信息直接写在配置文件中，但这存在安全风险。以下是推荐的安全配置最佳实践。

### 敏感信息加密与环境变量注入

当前配置中，`accessKey` 和 `secretKey` 等敏感信息明文存储在 `application-docker.yml` 中，这不符合安全最佳实践。建议采用以下改进措施：

1. **使用环境变量**: 将敏感信息通过环境变量注入，而非硬编码在配置文件中。例如：
   ```yaml
   minio:
     accessKey: ${MINIO_ACCESS_KEY}
     secretKey: ${MINIO_SECRET_KEY}
   ```

2. **配置加密**: 使用 Spring Cloud Config Server 或 Jasypt 等工具对配置文件中的敏感信息进行加密。

3. **密钥管理服务**: 在生产环境中，使用 AWS KMS、Hashicorp Vault 等专业密钥管理服务来存储和管理密钥。

4. **最小权限原则**: 为 MinIO 访问密钥分配最小必要的权限，避免使用管理员密钥。

通过这些措施，可以显著提高文件存储服务的安全性，防止敏感信息泄露。
