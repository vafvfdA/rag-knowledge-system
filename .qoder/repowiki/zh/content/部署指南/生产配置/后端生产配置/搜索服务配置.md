## Elasticsearch集群配置

`application-docker.yml` 配置文件中定义了Elasticsearch集群的连接参数，这些参数通过Spring的`@Value`注解读取并注入到`EsConfig`配置类中。

**配置项说明：**
- **主机地址 (host)**: `localhost` - 指定Elasticsearch服务器的主机名或IP地址。
- **端口号 (port)**: `9200` - 指定Elasticsearch的HTTP端口。
- **协议 (scheme)**: `http` - 指定与Elasticsearch通信的协议，开发环境使用HTTP，生产环境通常使用HTTPS。
- **用户名 (username)**: `elastic` - Elasticsearch的默认管理员用户名。
- **密码 (password)**: `PaiSmart2025` - 用于连接Elasticsearch的密码。

```yaml
elasticsearch:
  host: localhost       # Elasticsearch主机地址
  port: 9200            # Elasticsearch端口号
  scheme: http          # 协议（http/https）
  username: elastic     # 安装后生成的默认用户
  password: PaiSmart2025
```

**环境差异对比：**
在 `application.yml`（主配置）和 `application-docker.yml`（Docker环境配置）中，Elasticsearch的配置存在差异，主要体现在安全协议上：
- `application.yml` 使用 `scheme: https`，强调生产环境的安全性。
- `application-docker.yml` 使用 `scheme: http`，适用于开发和测试环境，简化配置。

## Elasticsearch客户端初始化

`EsConfig` 类负责初始化Elasticsearch客户端，它使用Elasticsearch官方的Java API客户端（版本8.10.0），通过`RestClient`构建底层HTTP连接，并创建高级的`ElasticsearchClient`实例。

### 初始化过程分析

1.  **配置注入**：`@Value` 注解从配置文件中读取 `host`, `port`, `scheme`, `username`, `password` 等属性。
2.  **构建RestClient**：使用 `RestClient.builder()` 方法，传入 `HttpHost` 对象来指定连接地址。
3.  **配置认证与安全**：
    *   如果提供了用户名和密码，会创建一个 `BasicCredentialsProvider` 并设置凭证。
    *   通过 `setHttpClientConfigCallback` 配置HTTP客户端，其中包含一个关键的安全设置：**忽略TLS证书验证**。代码中使用 `SSLContexts.custom().loadTrustMaterial(null, (chain, authType) -> true)` 来信任所有证书。这在开发环境中是可接受的，但在生产环境中是**严重的安全风险**，应使用有效的CA证书。
4.  **创建高级客户端**：将构建好的 `RestClient` 和 `JacksonJsonpMapper`（用于JSON序列化）传递给 `RestClientTransport`，最终创建 `ElasticsearchClient` 实例并作为Spring Bean返回。

```java
@Configuration
public class EsConfig {

    @Value("${elasticsearch.host}")
    private String host;
    @Value("${elasticsearch.port}")
    private int port;
    @Value("${elasticsearch.scheme:https}")
    private String scheme;
    @Value("${elasticsearch.username:elastic}")
    private String username;
    @Value("${elasticsearch.password:changeme}")
    private String password;

    @Bean
    public ElasticsearchClient elasticsearchClient() {
        RestClientBuilder builder = RestClient.builder(new HttpHost(host, port, scheme));

        if (username != null && !username.isEmpty()) {
            BasicCredentialsProvider credsProvider = new BasicCredentialsProvider();
            credsProvider.setCredentials(AuthScope.ANY, new UsernamePasswordCredentials(username, password));
            builder.setHttpClientConfigCallback(httpClientBuilder -> {
                try {
                    SSLContext sslContext = SSLContexts.custom()
                            .loadTrustMaterial(null, (X509Certificate[] chain, String authType) -> true) // 忽略证书验证
                            .build();
                    httpClientBuilder.setSSLContext(sslContext);
                    httpClientBuilder.setSSLHostnameVerifier(NoopHostnameVerifier.INSTANCE);
                } catch (Exception e) {
                    // ignore
                }
                return httpClientBuilder.setDefaultCredentialsProvider(credsProvider);
            });
        }

        RestClient restClient = builder.build();
        ElasticsearchTransport transport = new RestClientTransport(restClient, new JacksonJsonpMapper());
        return new ElasticsearchClient(transport);
    }
}
```

**关于连接超时：**
在 `EsConfig.java` 的代码中，**没有显式地设置连接超时（connection timeout）或套接字超时（socket timeout）**。这意味着客户端将使用Apache HttpClient的默认超时值。这可能导致在Elasticsearch服务器无响应时，客户端长时间阻塞。最佳实践是显式配置超时，例如：
```java
builder.setRequestConfigCallback(requestConfigBuilder -> 
    requestConfigBuilder
        .setConnectTimeout(5000) // 连接超时5秒
        .setSocketTimeout(30000) // 套接字超时30秒
);
```

## 索引配置与文档结构

### 索引命名与初始化

*   **索引名称 (Index Name)**: 所有文档都存储在名为 `knowledge_base` 的索引中。这个名称在 `EsIndexInitializer.java` 和 `ElasticsearchService.java` 中被硬编码。
*   **初始化过程**: `EsIndexInitializer` 类实现了 `CommandLineRunner` 接口，确保在Spring Boot应用启动时自动执行。其流程如下：
    1.  检查 `knowledge_base` 索引是否存在。
    2.  如果不存在，则从 `classpath:es-mappings/knowledge_base.json` 加载映射文件。
    3.  调用Elasticsearch API创建索引，并应用加载的映射。

```java
@Component
public class EsIndexInitializer implements CommandLineRunner {
    @Autowired
    private ElasticsearchClient esClient;

    @Value("classpath:es-mappings/knowledge_base.json")
    private org.springframework.core.io.Resource mappingResource;

    @Override
    public void run(String... args) throws Exception {
        initializeIndex();
    }

    private void initializeIndex() throws Exception {
        BooleanResponse existsResponse = esClient.indices().exists(ExistsRequest.of(e -> e.index("knowledge_base")));
        if (!existsResponse.value()) {
            createIndex();
        } else {
            logger.info("索引 'knowledge_base' 已存在");
        }
    }

    private void createIndex() throws Exception {
        String mappingJson = new String(Files.readAllBytes(mappingResource.getFile().toPath()), StandardCharsets.UTF_8);
        CreateIndexRequest createIndexRequest = CreateIndexRequest.of(c -> c
                .index("knowledge_base")
                .withJson(new StringReader(mappingJson))
        );
        esClient.indices().create(createIndexRequest);
        logger.info("索引 'knowledge_base' 已创建");
    }
}
```

**分片与副本数量：**
在 `knowledge_base.json` 映射文件和 `EsIndexInitializer.java` 的创建逻辑中，**没有显式地指定分片（shards）和副本（replicas）的数量**。这意味着Elasticsearch将使用其默认设置（通常是1个主分片和1个副本）。对于生产环境，应根据数据量和性能需求在创建索引时明确配置这些参数。

### 文档结构（映射）

`knowledge_base.json` 文件定义了 `knowledge_base` 索引的文档结构（映射），核心字段如下：

```json
{
  "mappings": {
    "properties": {
      "fileMd5": { "type": "keyword" },
      "chunkId": { "type": "integer" },
      "textContent": { "type": "text", "analyzer": "standard" },
      "vector": { "type": "dense_vector", "dims": 2048, "index": true, "similarity": "cosine" },
      "modelVersion": { "type": "keyword" },
      "userId": { "type": "keyword" },
      "orgTag": { "type": "keyword" },
      "isPublic": { "type": "boolean" }
    }
  }
}
```

**关键字段解析：**
- **`textContent` (文本内容)**: 类型为 `text`，使用 `standard` 分析器。这是进行全文检索（BM25算法）的基础字段。
- **`vector` (向量)**: 类型为 `dense_vector`，维度（dims）为2048。该字段用于存储文档块的向量表示，支持KNN（K-Nearest Neighbors）近似最近邻搜索，是实现语义搜索的核心。`similarity: "cosine"` 指定了使用余弦相似度来计算向量间的距离。
- **`fileMd5` 和 `chunkId`**: 用于唯一标识一个文档块，`fileMd5` 标识原始文件，`chunkId` 标识文件内的文本块。
- **`userId`, `orgTag`, `isPublic`**: 这些是权限过滤字段，用于在搜索时根据用户身份和组织标签进行访问控制。

## 搜索请求配置

### 混合搜索逻辑

`HybridSearchService` 类实现了RAG系统中关键的混合搜索功能，结合了向量相似度搜索（语义）和文本匹配搜索（关键词）。

1.  **向量生成**: 使用 `EmbeddingClient` 将用户查询 `query` 转换为一个2048维的向量。
2.  **KNN召回**: 使用 `s.knn()` 执行向量搜索，从 `vector` 字段中召回与查询向量最相似的前 `recallK` 个文档（`recallK = topK * 30`）。
3.  **查询过滤**: 使用 `s.query()` 中的 `bool` 查询，确保召回的文档必须包含查询关键词（`must` 条件），并应用权限过滤（`filter` 条件）。
4.  **重打分 (Rescore)**: 这是混合搜索的关键步骤。系统使用 `s.rescore()` 对KNN召回的前 `recallK` 个结果进行重排序。
    *   `queryWeight(0.2d)`: 保留20%的原始KNN分数。
    *   `rescoreQueryWeight(1.0d)`: 给予重打分查询100%的权重。
    *   重打分查询本身是一个 `match` 查询，使用BM25算法在 `textContent` 字段上进行关键词匹配。
    *   **效果**: 最终排名由BM25（关键词相关性）主导，但保留了KNN（语义相关性）的部分影响，实现了语义和关键词的平衡。
5.  **结果返回**: 使用 `s.size(topK)` 限制最终返回的结果数量。

```java
s.knn(kn -> kn
    .field("vector")
    .queryVector(queryVector)
    .k(recallK)
    .numCandidates(recallK)
);
s.query(q -> q.bool(b -> b
    .must(mst -> mst.match(m -> m.field("textContent").query(query)))
    // ... 权限过滤
));
s.rescore(r -> r
    .windowSize(recallK)
    .query(rq -> rq
        .queryWeight(0.2d)
        .rescoreQueryWeight(1.0d)
        .query(rqq -> rqq.match(m -> m
            .field("textContent")
            .query(query)
            .operator(Operator.And)
        ))
    )
);
s.size(topK);
```

### API端点与参数

`SearchController` 提供了 `/api/search/hybrid` 这个REST API端点。

*   **分页大小 (topK)**: 通过 `@RequestParam(defaultValue = "10") int topK` 参数控制。前端默认值为10，用户可在1-50范围内调整。此参数直接传递给 `HybridSearchService.search()` 方法，控制最终返回的文档数量。
*   **超时配置**: 在 `SearchController.java` 和 `HybridSearchService.java` 的代码中，**没有发现对Elasticsearch搜索请求本身设置超时**。搜索操作的超时依赖于底层HTTP客户端（`RestClient`）的配置，而如前所述，该配置并未在 `EsConfig` 中显式设置。这可能导致搜索请求在极端情况下长时间挂起。
