## 引言
本文档旨在全面阐述生产环境中敏感信息（如数据库密码、JWT密钥、MinIO凭证等）的安全保护策略。基于对PaiSmart项目的深入分析，文档将评估当前敏感信息的存储方式，提出使用Jasypt进行属性加密的解决方案，并探讨通过环境变量注入配置的最佳实践。同时，文档将分析日志脱敏机制，防止敏感信息在日志中泄露，并提出一系列安全加固建议，包括强制启用HTTPS、配置HSTS、设置安全响应头以及定期轮换密钥等措施，以全面提升系统的安全性。

## 敏感信息存储现状分析

通过对`application-docker.yml`等配置文件的分析，发现当前项目存在严重的敏感信息明文存储问题。以下是在`application-docker.yml`文件中识别出的关键敏感信息：

```yaml
spring:
  datasource:
    password: PaiSmart2025 # 数据库密码明文存储
  data:
    redis:
      password: PaiSmart2025 # Redis密码明文存储
minio:
  accessKey: aHZtaNPEzLcERm9uCY2F
  secretKey: 5lhyVdLKfMfYXtmdVsulLYpgCRx5EO1EfIWAtJdq # MinIO密钥明文存储
jwt:
  secret-key: "PXrQbuCwXwOZzkML/Vm2S5rSwt1iybvmKtGDzVEu+Hc=" # JWT密钥明文存储
admin:
  password: admin123 # 管理员密码明文存储
elasticsearch:
  password: PaiSmart2025 # Elasticsearch密码明文存储
deepseek:
  api:
    key: sk-cfd92d6071d246f9a4c8f941a735c271 # DeepSeek API Key明文存储
embedding:
  api:
    key: b22d5f8a-a3cf-4c95-bb74-e05c45211474 # 火山引擎API Key明文存储
```

**问题分析**：
1.  **明文存储风险**：所有敏感信息均以明文形式直接写入YAML配置文件，一旦配置文件泄露（如通过代码仓库、服务器文件系统），攻击者可立即获取所有系统凭证。
2.  **缺乏加密机制**：项目当前未集成任何配置属性加密框架（如Jasypt），无法对配置文件中的敏感值进行加密。
3.  **硬编码问题**：敏感信息被硬编码在配置文件中，导致其与代码库耦合，难以在不同环境（开发、测试、生产）间安全地进行切换。

## Jasypt属性加密方案

### Jasypt集成必要性
Jasypt（Java Simplified Encryption）是一个为Java项目提供简单加密功能的库，特别适用于对Spring Boot应用的配置文件进行加密。它允许开发者将敏感信息（如密码、密钥）加密后存储在`application.yml`中，运行时由Jasypt自动解密。

### 当前项目状态评估
通过搜索项目依赖文件`pom.xml`，确认项目**未集成**Jasypt库：
```xml
<!-- pom.xml中未找到jasypt相关依赖 -->
```
此外，代码库中也未发现任何与Jasypt相关的配置或使用痕迹。

### 推荐实施步骤
为解决敏感信息明文存储问题，强烈建议集成Jasypt。以下是实施步骤：

1.  **添加Maven依赖**：
    在`pom.xml`中添加Jasypt Spring Boot Starter依赖。
    ```xml
    <dependency>
        <groupId>com.github.ulisesbocchio</groupId>
        <artifactId>jasypt-spring-boot-starter</artifactId>
        <version>3.0.5</version>
    </dependency>
    ```

2.  **生成主加密密钥（Master Password）**：
    该密钥用于解密配置文件中的所有加密值，必须严格保密，**绝不能**存入代码库。
    ```bash
    # 使用Jasypt提供的命令行工具加密一个测试字符串
    java -cp jasypt-1.9.3.jar org.jasypt.intf.cli.JasyptPBEStringEncryptionCLI input="test" password=yourMasterPassword algorithm=PBEWithMD5AndDES
    ```
    输出结果如：`ENC(2a3f1c4e5b6d7e8f)`。其中`yourMasterPassword`即为主密钥。

3.  **加密敏感信息并修改配置**：
    使用上一步生成的主密钥，将`application-docker.yml`中的敏感值逐一加密。
    ```yaml
    spring:
      datasource:
        password: ENC(加密后的数据库密码) # 例如: ENC(uO12j3p4q5r6s7t8)
      data:
        redis:
          password: ENC(加密后的Redis密码)
    minio:
      secretKey: ENC(加密后的MinIO密钥)
    jwt:
      secret-key: ENC(加密后的JWT密钥)
    # ... 其他敏感字段同理
    ```

4.  **通过环境变量注入主密钥**：
    在启动应用时，通过环境变量`JASYPT_ENCRYPTOR_PASSWORD`传入主密钥，实现密钥的独立管理。
    ```bash
    # Docker启动示例
    docker run -e JASYPT_ENCRYPTOR_PASSWORD=yourMasterPassword -e SPRING_PROFILES_ACTIVE=docker your-app-image
    ```

## 环境变量注入最佳实践

为避免在代码或配置文件中硬编码任何敏感信息，应采用环境变量注入作为核心安全实践。

### 当前项目状态
项目代码中未发现通过`@Value("${ENV_VAR}")`或`System.getenv()`等方式从环境变量读取敏感配置的逻辑。所有配置均直接从YAML文件加载。

### 推荐实施策略
1.  **移除配置文件中的敏感值**：
    将`application-docker.yml`中所有敏感字段的值替换为环境变量占位符。
    ```yaml
    spring:
      datasource:
        url: ${DB_URL:jdbc:mysql://localhost:3306/PaiSmart}
        username: ${DB_USERNAME:root}
        password: ${DB_PASSWORD} # 从环境变量读取
      data:
        redis:
          password: ${REDIS_PASSWORD}
    minio:
      accessKey: ${MINIO_ACCESS_KEY}
      secretKey: ${MINIO_SECRET_KEY}
    jwt:
      secret-key: ${JWT_SECRET_KEY}
    # ... 其他字段
    ```

2.  **在部署环境中设置环境变量**：
    在Docker、Kubernetes或云平台的部署配置中，通过安全的方式（如Secrets、Parameter Store）设置这些环境变量。
    ```yaml
    # Docker Compose 示例
    version: '3'
    services:
      pai-smart:
        image: your-app-image
        environment:
          - DB_PASSWORD=PaiSmart2025
          - REDIS_PASSWORD=PaiSmart2025
          - MINIO_ACCESS_KEY=aHZtaNPEzLcERm9uCY2F
          - MINIO_SECRET_KEY=5lhyVdLKfMfYXtmdVsulLYpgCRx5EO1EfIWAtJdq
          - JWT_SECRET_KEY=PXrQbuCwXwOZzkML/Vm2S5rSwt1iybvmKtGDzVEu+Hc=
        # ... 其他配置
    ```

3.  **结合Jasypt使用（推荐）**：
    最佳实践是将环境变量注入与Jasypt加密结合。即，使用环境变量注入Jasypt的主密钥，而具体的敏感信息（如数据库密码）则在配置文件中以加密形式存储。
    ```bash
    # 启动命令
    java -jar app.jar --jasypt.encryptor.password=${JASYPT_MASTER_PASSWORD}
    ```

**优势**：
- **解耦配置与代码**：敏感信息与代码库完全分离。
- **环境隔离**：不同环境（dev, test, prod）可以使用不同的环境变量值。
- **增强安全性**：即使配置文件泄露，攻击者也无法获取明文密码。

## 日志脱敏与敏感信息掩码

### 当前日志脱敏机制分析
通过对`LogUtils.java`和`logback-spring.xml`的分析，发现项目当前**缺乏**对敏感信息的日志脱敏处理。

1.  **LogUtils.java分析**：
    `LogUtils`类提供了多种日志记录方法（如`logBusiness`, `logUserOperation`），但其方法签名和实现中均未包含对敏感数据（如密码、密钥）的自动掩码逻辑。日志消息中的参数会直接被格式化输出。
    ```java
    public static void logBusiness(String operation, String userId, String message, Object... args) {
        // args中的敏感信息（如密码）会被直接格式化到日志中
        BUSINESS_LOGGER.info("[{}] [用户:{}] {}", operation, userId, formatMessage(message, args));
    }
    ```
    **结论**：如果业务代码在日志中打印了包含密码的用户对象或请求体，这些信息将以明文形式记录在日志文件中。

2.  **logback-spring.xml分析**：
    该文件配置了日志的输出格式、级别和文件滚动策略。虽然为不同环境（dev/prod）设置了不同的日志级别（生产环境为INFO/WARN），但**没有配置任何日志过滤器（Filter）来实现敏感信息的脱敏**。
    ```xml
    <!-- 缺少类似以下的脱敏过滤器配置 -->
    <filter class="ch.qos.logback.classic.filter.EvaluatorFilter">
        <evaluator>
            <expression>message.contains("password") || message.contains("key")</expression>
        </evaluator>
        <onMatch>NEUTRAL</onMatch>
    </filter>
    ```

### 推荐日志脱敏实现方案
1.  **增强LogUtils工具类**：
    在`LogUtils`中增加对敏感字段的自动掩码功能。
    ```java
    public class LogUtils {
        // 敏感字段列表
        private static final Set<String> SENSITIVE_FIELDS = Set.of("password", "secret", "key", "token", "pwd");

        /**
         * 安全地格式化消息，对已知的敏感字段进行掩码
         */
        private static String sanitizeMessage(String message, Object... args) {
            if (args == null || args.length == 0) return message;
            try {
                String formatted = String.format(message, args);
                // 简单的掩码处理：将password=xxx替换为password=***
                for (String field : SENSITIVE_FIELDS) {
                    formatted = formatted.replaceAll("(?i)" + field + "=[^,&\\s]+", field + "=***");
                }
                return formatted;
            } catch (Exception e) {
                return message + " [格式化参数失败: " + e.getMessage() + "]";
            }
        }
    }
    ```

2.  **使用Logback MDC进行上下文脱敏**：
    利用MDC（Mapped Diagnostic Context）在日志中传递用户ID等非敏感信息，避免在日志消息中直接拼接用户凭证。

3.  **配置Logback过滤器**：
    在`logback-spring.xml`中添加自定义过滤器，对包含敏感关键词的日志进行拦截或修改。
    ```xml
    <filter class="com.yizhaoqi.smartpai.utils.SensitiveDataFilter" />
    ```

## 安全加固建议

### 1. 强制启用HTTPS
当前`SecurityConfig.java`中未配置任何强制HTTPS的规则，应用默认通过HTTP提供服务，存在中间人攻击（MITM）风险。

**建议**：
- 在`SecurityConfig`中添加`requiresChannel()`配置，将所有HTTP请求重定向到HTTPS。
    ```java
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            // ... 其他配置
            .requiresChannel(channel -> channel
                .requestMatchers("/api/**").requiresSecure() // API端点强制HTTPS
                .anyRequest().requiresSecure() // 所有请求强制HTTPS
            );
        return http.build();
    }
    ```

### 2. 配置HSTS（HTTP Strict Transport Security）
HSTS是一种安全策略机制，它告诉浏览器只能通过HTTPS与服务器通信，有效防止SSL剥离攻击。

**建议**：
- 在`SecurityConfig`中启用HSTS。
    ```java
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            // ... 其他配置
            .headers(headers -> headers
                .httpStrictTransportSecurity(hsts -> hsts
                    .maxAgeInSeconds(31536000) // 一年
                    .includeSubDomains(true)
                    .preload(true)
                )
            );
        return http.build();
    }
    ```

### 3. 设置安全响应头
当前`SecurityConfig`未配置任何安全响应头。

**建议**：
- 在`SecurityConfig`中添加关键安全响应头。
    ```java
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            // ... 其他配置
            .headers(headers -> headers
                .contentSecurityPolicy(csp -> csp
                    .policyDirectives("default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data:;")
                )
                .xssProtection(xss -> xss.headerValue(XXssProtectionHeaderWriter.HeaderValue.ENABLED_MODE_BLOCK))
                .contentTypeOptions(contentType -> contentType)
                .frameOptions(frameOptions -> frameOptions.deny()) // 防止点击劫持
                .cacheControl(cache -> cache) // 控制缓存
            );
        return http.build();
    }
    ```
    - **X-Content-Type-Options**: 设置为`nosniff`，防止浏览器进行MIME类型嗅探，避免XSS攻击。
    - **X-Frame-Options**: 设置为`DENY`或`SAMEORIGIN`，防止页面被嵌入到`<frame>`或`<iframe>`中，抵御点击劫持攻击。

### 4. 定期轮换密钥
长期使用同一密钥会增加泄露风险。

**建议**：
- **JWT密钥轮换**：项目提供了`GenerateJwtKey.java`工具类，可用于生成新的JWT密钥。应建立定期（如每季度）轮换JWT密钥的流程，并确保新旧密钥在一段时间内共存，以平滑过渡。
- **数据库/服务凭证轮换**：定期更新数据库、Redis、MinIO等服务的密码和密钥。
- **API Key轮换**：定期轮换DeepSeek、火山引擎等第三方服务的API Key。

## 结论
当前PaiSmart项目在敏感信息保护方面存在显著的安全隐患，主要表现为敏感信息在配置文件中明文存储、缺乏日志脱敏机制以及未实施关键的Web安全加固措施。为提升系统安全性，必须立即采取以下行动：
1.  **立即集成Jasypt**，对`application-docker.yml`中的所有敏感属性进行加密。
2.  **全面采用环境变量注入**，将敏感信息的管理从代码库转移到安全的部署环境中。
3.  **增强日志脱敏能力**，修改`LogUtils`并配置Logback过滤器，防止敏感信息在日志中泄露。
4.  **实施安全加固**，在`SecurityConfig`中强制启用HTTPS、配置HSTS和关键安全响应头。
5.  **建立密钥轮换制度**，定期更新JWT密钥及其他服务凭证。

通过实施上述措施，可以构建一个纵深防御的安全体系，有效保护系统的敏感信息，降低安全风险。
